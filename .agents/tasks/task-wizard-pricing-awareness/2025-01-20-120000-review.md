# Pricing awareness in the Synergy Wizard

The wizard gains a budget filter and cost estimation layer. The `Connector` interface is extended with four optional pricing fields, all 166 connectors receive empty placeholders, scoring grows a `maxMonthlyCost` parameter with 10% tolerance, and the UI adds a budget input, per-card pricing display, and an estimated-cost footer. A new `load-connectors.ts` module wraps the connector source to prepare for future dynamic loading from AsyncStorage.

Watch for: The budget filter is structurally inert today because every connector ships with `pricingTiers: []` — no connector will ever be filtered or priced until real data arrives (confirmed). The async `loadConnectors()` call is fire-and-forget on mount; its result is never consumed by the scoring path, which calls `getConnectorsSync()` instead (confirmed). Price parsing assumes a single numeric substring per tier string, which will break on formats like "$0.002/1K tokens" (likely).

## High-level view

The schema extension is cleanly additive — four optional fields, no breaking change. The placeholders make the feature dormant until a data pipeline fills in real values. How the wizard will know which service costs what is explicitly deferred (the "RAPIER DB export" TODO), meaning this feature cannot be validated end-to-end until that dependency materializes.

The scoring engine gains a budget ceiling with 10% over-budget tolerance (so a 30 EUR budget still surfaces 33 EUR connectors that fit the workflow better), plus a flat +1 score bonus for `apiAccess === true`. The filter is a no-op today because empty `pricingTiers` arrays skip the budget check — a fail-open that will need a product decision once data flows in incrementally.

The data loader introduces an async/sync split that doesn't connect: `loadConnectors()` is called on mount but its result is never cached for `getConnectorsSync()` to return. Scoring always uses the bundled array regardless of AsyncStorage state.

The UI cost estimation duplicates the price-parsing regex from `scoring.ts` and displays `~$X/mo` in dollars regardless of locale — conflicting with the app's German labels and Euro examples.

## Async/sync connector split

`load-connectors.ts` exports two functions with fundamentally different data sources. `loadConnectors()` checks AsyncStorage and returns the result; `getConnectorsSync()` ignores AsyncStorage entirely and returns the compile-time array. The wizard calls `void loadConnectors()` on mount — a fire-and-forget pattern that warms nothing, because no module-level cache is set from the async result.

Even if a user populates `anvil_connectors_v1` in AsyncStorage (via a future RAPIER sync), scoring will still use the bundled connectors. The fix:

```typescript
let _cached: Connector[] | null = null;

export async function loadConnectors(): Promise<Connector[]> {
  // ... existing logic ...
  _cached = result;
  return result;
}

export function getConnectorsSync(): Connector[] {
  return _cached ?? CONNECTORS;
}
```

Without this, the async path is dead code.

## Budget filter fail-open behavior

Connectors with `pricingTiers: []` (currently all of them) skip the budget check because `tiers.length > 0` is false. This means paid connectors without pricing data are never filtered by budget. Once real data starts flowing incrementally, any connector that hasn't been populated yet will bypass the budget constraint — a fail-open behavior. Whether this is desirable (show everything we can't price) or a bug (should treat unknown-price as "possibly over budget") is a product decision worth documenting.

## Price parsing fragility

`parseCheapestPrice` uses `/[\d]+(?:[.,]\d+)?/` to extract the first numeric value from a freeform string. It would misparse:
- "1,000 req/day" → comma treated as decimal separator, yielding `1.0` instead of `1000`
- Tier strings that mix price with limits like "Free (1000 req/day)" → extracts `1000` as if it were a price

Since the schema already has a separate `limits` field on each tier, the `price` field should be constrained to a pure price string. Adding a `numericPrice?: number` field to the tier schema would eliminate parsing entirely and make the budget filter robust.

## UI cost estimation

The display hardcodes `~$${estimatedCost}/mo` which conflicts with the German-language UI. The cost is a sum of the cheapest tier from each stage, but doesn't consult the `limits` field — a free tier with 100 requests/day might not be viable for a trigger that fires hourly, making the estimate misleading when it says "$0" for a stage that will actually require a paid tier to function.
