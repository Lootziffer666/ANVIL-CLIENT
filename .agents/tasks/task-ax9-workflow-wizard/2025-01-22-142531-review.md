# Synergy Workflow Wizard tab with local connector scoring

A new "Wizard" tab provides a self-contained automation-pipeline generator running entirely on-device. The user enters their current stack, a goal phrase, and a budget tier; after a 500ms debounce the app scores 200 bundled connectors and fills four pipeline stages (Trigger, LLM Processor, Action, MCP Context). The implementation adds ~3.5k lines across a data file, a scoring module, a UI screen, and minor wiring in the tab layout and icon map.

**Watch for:** The scoring algorithm's stage-assignment logic causes nearly every connector to compete in the MCP Context slot (confirmed — 195 of 200 qualify), diluting what should be the most selective stage. The `topCategory` shown for MCP Context hits is always a routing keyword like "trigger" rather than the domain category the user would find informative (confirmed). A dead-code function `getStage` is present alongside its replacement `getStages` (confirmed).

## High-level view

The connector data is a 3 104-line TypeScript constant exporting 200 typed objects. The 34 connectors marked `dead: true` ship in the bundle and are filtered only at scoring time — stripping them from the constant would save ~500 lines.

Scoring combines keyword overlap between the user's goal tokens and each connector's `aiNotes`/`categories` with a +3 stack-match bonus. The percentage normalization divides by `goalTokens.length + 3`, which means the denominator shifts as the user types more words — a connector's displayed score drops if the user adds irrelevant terms even when the match quality is identical.

Stage assignment uses `getStages` to map a connector into one or more pipeline slots based on its categories containing the literal strings "trigger", "action", "llm", "ai", or "ml". Because 195 of 200 connectors have at least one non-routing category (e.g. "email", "crm"), nearly all qualify for MCP Context in addition to their primary slot. The `topCategory` display for MCP Context then picks the first category that passes a truthy filter — which is always the first element in the array, often "trigger" or "action", not the domain category the user would expect to see.

The icon-symbol change narrows the MAPPING type from `as IconMapping` to `satisfies Partial<IconMapping>`. This changes the type contract: any future icon addition requires updating MAPPING first — previously the cast allowed any SF Symbol name at the type level.

## Stage assignment and the MCP Context collapse

The `getStages` function assigns a connector to multiple pipeline stages simultaneously. The relevant logic:

```typescript
if (stages.length === 0 || lower.some((c) => !['trigger', 'action', 'llm', 'ai', 'ml'].includes(c))) {
  stages.push('mcpContext');
}
```

Since every connector except five pure-LLM entries has at least one domain category (e.g. "email", "crm", "database"), 195 connectors end up in the `mcpContext` pool. This means the MCP Context card shows whichever connector happens to score highest globally — typically the same connector that already won one of the other three slots. A user looking at the pipeline sees the same tool repeated or a result indistinguishable from the Action card.

The `topCategory` display compounds this. For `mcpContext`, the `.find()` callback effectively returns `true` for every category, so it picks the first element in `connector.categories`. For the 128 connectors tagged both "trigger" and "action", the array often starts with a routing keyword rather than the domain label ("email", "crm") the user would find meaningful.

## Score normalization drift

The denominator for percentage calculation is `goalTokens.length + 3`. With a one-word goal like "email", that's 4, so a single keyword match yields 25%. With a five-word goal "prd creation code review automation", the denominator is 8, so the same single keyword match yields 12%. The connector didn't get worse; the user just typed more. This makes scores feel unstable as the user types, undermining the live-update UX.
