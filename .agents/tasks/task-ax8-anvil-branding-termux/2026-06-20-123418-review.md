# Anvil branding and Termux server integration

The app's tab bar drops the "Chat" label in favor of "Forge" (with a hammer icon) and renames the workspace screen title to "Anvil Workspace". A new `lib/termux/` module provides Android intent-based helpers for detecting Termux, launching a LiteLLM proxy inside it, and checking server health on localhost:4000. A settings section (`TermuxSection`) exposes these controls with start/stop buttons and status readouts. The CI workflow is separately reworked for faster ARM64-only Android builds.

**Watch for:** `isTermuxInstalled()` actually launches Termux as a side-effect of checking if it exists (confirmed). The `com.termux.permission.RUN_COMMAND` permission is not declared in `app.config.ts`, so the RUN_COMMAND intent will fail at runtime on Android 11+ (confirmed). No `<queries>` element declares the Termux package, which means `startActivityAsync` may silently fail on Android 11+ package-visibility rules (confirmed).

## High-level view

The Termux integration introduces a fire-and-forget lifecycle for the local Bellows server. The app launches an intent, assumes it worked, and then polls `localhost:4000/health` to determine status. There is no persistent connection, no process-ID tracking, and no way to know if the server is still the one *this app* launched. That's an acceptable first pass for a local-dev tool, but the detection mechanism (`isTermuxInstalled`) has a destructive side-effect, and the Android permission setup is incomplete.

The CI workflow change reduces build scope to ARM64-only, adds several cache layers, and triggers on pull requests to `main`. It's a separate concern bundled into the same branch.

## Detection via intent side-effect

`isTermuxInstalled()` calls `IntentLauncher.startActivityAsync('android.intent.action.MAIN', { packageName: TERMUX_PACKAGE })`. If Termux is installed this actually *opens Termux* on screen. The function is called on mount inside `TermuxSection`'s `useEffect`, so every time the user navigates to Settings, Termux will foreground itself.

A non-invasive check would use `Linking.canOpenURL('com.termux://')` (if Termux registers that scheme) or a thin native module calling `PackageManager.getPackageInfo(TERMUX_PACKAGE, 0)`. Both query without launching.

## Android 11+ package visibility and permission gap

Starting with Android 11 (API 30), apps cannot discover other installed packages unless they declare them in a `<queries>` block or hold `QUERY_ALL_PACKAGES`. Without this, `startActivityAsync` targeting `com.termux` may throw `ActivityNotFoundException` even when Termux is present.

The `com.termux.permission.RUN_COMMAND` permission must also be declared in the calling app's manifest:

```xml
<uses-permission android:name="com.termux.permission.RUN_COMMAND"/>
```

Neither is present in `app.config.ts`. Both require either an Expo config plugin that injects manifest entries or manual `android/app/src/main/AndroidManifest.xml` edits. Without them the entire RUN_COMMAND flow is non-functional on target devices.

## Server lifecycle is fire-and-forget

After tapping "Start Server", the UI returns to idle and "Bellows Server" still reads "Not reachable" until the user manually taps "Refresh Status" â€” there is no automatic poll after launch. A retry loop (e.g., every 2s for 20s) would close the gap.

`stopBellowsServer()` sends `pkill -f litellm || true`. If the user has another LiteLLM instance running for a different purpose, this kills it too. The `|| true` mask means a failed kill is indistinguishable from success â€” the component optimistically sets `serverHealthy(false)` regardless.

## CI workflow rewrite (bundled unrelated change)

The GitHub Actions workflow adds a `pull_request` trigger, drops the duplicate diagnostics upload step, adds `node_modules` + Android build caches, pins builds to ARM64-only via `GRADLE_ARGS` and `ORG_GRADLE_PROJECT_reactNativeArchitectures`, and writes Gradle daemon/parallel/caching properties at build time. Unrelated to the branding or Termux feature and should ship separately.
